# 开发文档 v2（智能构图摄影辅助系统：photo_backend + VLM）

更新时间：2026-03-16

本文档面向“开发/联调/交接”，目标是把**每个功能模块做什么、为什么这么做、代码里怎么实现、数据怎么流转**讲清楚。

如果你只想看“每天开工怎么启动”，请直接看同目录的《启动说明.md》。

## 1. 项目整体说明（你现在跑起来的形态是什么）

### 1.1 当前联调形态（非常重要）

你们目前的真实运行边界是：

- **服务器（Linux）**：运行 Go 后端（Gin），对外提供统一 REST API，默认端口 `8080`
- **本地（Windows，有 GPU）**：运行 Python VLM 推理服务（FastAPI/Uvicorn），端口 `127.0.0.1:8000`
- **连接方式**：本地用 SSH 反向端口转发把本地 `127.0.0.1:8000` 映射到服务器 `127.0.0.1:8000`

因此服务器上的 Go 后端在调用“拍摄建议/姿势引导”时，其实是在访问：

`http://127.0.0.1:8000/vlm/infer`（看起来是服务器本机，其实是穿隧道到你本地 GPU）

> 这套形态用于开发联调非常好，但生产长期运行应改成：Python VLM 部署在一台 GPU 机器上，服务器 Go 直接走内网访问，不再依赖开发机与 SSH 隧道。

### 1.2 两个推理模块分别在哪跑

- **经典构图识别（ONNX）**：在 Go 后端所在机器执行（服务器端）
  - 模型：`models/comp_model.onnx`
  - 推理框架：`onnxruntime_go`

- **拍摄建议/姿势引导（VLM 多模态大模型）**：在 Python VLM 服务所在机器执行（你本地 Windows GPU）
  - 模型：`Qwen/Qwen2-VL-2B-Instruct`
  - 推理框架：HuggingFace `transformers`

## 2. 代码结构与分层（Go 后端）

你们 Go 后端遵循典型分层：

- `router/`：注册路由与中间件（CORS、静态资源、上传大小）
- `controller/`：接收请求、参数校验、统一返回格式、调用 Service
- `service/`：业务逻辑编排（数据库读写、排序/推荐、转发到 Python VLM 等）
- `dao/`：初始化数据库连接（GORM），AutoMigrate
- `model/`：数据库表结构定义（GORM Model）
- `static/`：静态资源根目录（模板、上传图片）

分层的核心价值：
- Controller 专注“HTTP 语义”和入参校验
- Service 专注“业务决策/算法”和对下游（DB、文件、外部服务）的组合
- DAO/Model 把数据结构稳定化，避免 SQL 到处散落

## 3. 全局配置与启动流程（Go 后端）

### 3.1 Go 后端入口：main.go 做了哪些事

入口文件：`main.go`

启动顺序：

1) `dao.InitDB()`：连接 MySQL + AutoMigrate
2) 配置 ONNX Runtime 动态库路径（Linux 上常见坑）：
   - 优先读取环境变量 `ONNXRUNTIME_SHARED_LIBRARY_PATH`
   - 如果没设置，会尝试探测一些常见路径（只在 Linux 下做兜底）
   - 然后调用 `ort.SetSharedLibraryPath(...)`
3) 初始化构图模型（ONNX）：
   - 模型路径来自环境变量 `COMPOSITION_MODEL_PATH`，默认 `models/comp_model.onnx`
   - 会做相对路径解析（兼容 systemd / nohup 时 cwd 不一致）
   - `service.InitCompositionService(modelPath)`
4) `router.SetupRouter()`：注册路由 + CORS + 静态资源
5) `PORT` 环境变量指定端口（默认 8080）并启动 Gin

### 3.2 关键环境变量（Go）

- `PORT`：服务端口，默认 `8080`
- `PHOTO_DB_DSN`：MySQL DSN，默认 `root:123456@tcp(127.0.0.1:3306)/photography_db?...`
- `GIN_MODE`：建议 `release`

构图（ONNX）相关：
- `COMPOSITION_MODEL_PATH`：ONNX 模型路径（默认 `models/comp_model.onnx`）
- `ONNXRUNTIME_SHARED_LIBRARY_PATH`：Linux 常需要显式指定 `libonnxruntime.so`

VLM（拍摄建议/姿势引导）相关：
- `VLM_API_URL`：完整 URL（优先级最高），例如 `http://127.0.0.1:8000/vlm/infer`
- `VLM_API_BASE_URL`：基础地址（会自动拼 `/vlm/infer`），例如 `http://127.0.0.1:8000`
- `VLM_HTTP_TIMEOUT_SECONDS`：Go→Python 转发超时（建议 180）

> systemd 常驻时，推荐把这些变量写进 `/etc/photo_backend.env` 并由 service 的 `EnvironmentFile` 加载（避免每次手动 export）。

## 4. 静态资源设计（static/）

### 4.1 为什么要用静态目录

你们的业务里有两类文件：

1) **模板资源**（属于“产品资源”，相对稳定）
- 路径：`static/templates/`
- 包含：每个模板目录下的 cover/example/overlay，以及全量的 `templates.json`

2) **用户上传图片**（属于“用户数据”，持续增长）
- 保存路径：`static/uploads/u{user_id}/YYYYMMDD/xxx.jpg`
- 对外访问统一走：`/static/...`

Go 在 `router.SetupRouter()` 里做了：

- `r.Static("/static", "./static")`

因此客户端拿到的图片 URL 是：

`http(s)://{host}/static/...`

### 4.2 templates.json 元数据（模板“三图 + 引导”）

文件：`static/templates/templates.json`

每一项定义一个模板：
- `template_id`：模板 ID（主键）
- `title`：展示标题
- `tags[]`：标签
- `hot`：运营热度（基础分）
- `cover/example/overlay`：三张图的相对路径
- `guides[]`：结构化引导文字 + 归一化坐标

guides 坐标采用 0~1 的归一化比例，便于适配不同屏幕比例：
- `x`：相对宽度比例
- `y`：相对高度比例

服务端会把这些路径统一转成 `/static/templates/...` 的 URL，并带 `?v=mtime` 做简单缓存版本。

## 5. 数据库设计（model/ + dao/）

### 5.1 数据库初始化与迁移

文件：`dao/db.go`

- 环境变量 `PHOTO_DB_DSN` 控制连接串
- 连接成功后执行 `AutoMigrate`：
  - `model.User`
  - `model.Material`
  - `model.Preference`
  - `model.TemplateFavorite`
  - `model.TemplateUsage`

优点：
- 开发环境无需手动建表
- 表结构与代码强一致

注意：
- AutoMigrate 只会做“安全的变更”（新增列/索引等），复杂迁移仍建议人工控制

### 5.2 核心表与用途

- `users`：用户账号
  - 密码只存 bcrypt hash（不返回给前端）

- `materials`：素材（草稿/作品）
  - `status=0` 草稿，`status=1` 作品
  - `template_id` 可选：记录作品是否使用了某个模板（推荐与统计信号）

- `preferences`：偏好
  - 当前主要用 `tags`（逗号分隔），作为推荐的重要信号

- `template_favorites`：收藏关系
  - 对 `(user_id, template_id)` 唯一约束，避免重复收藏

- `template_usages`：模板使用量统计
  - 用 upsert 自增 `used_count`，用于热门/推荐排序与 UI 展示

## 6. API 与模块原理（Go 后端）

统一返回格式：

```json
{"code":200,"msg":"ok","data":{}}
```

> 目前大多数接口采用“HTTP 200 + 业务 code”，方便客户端统一处理。

### 6.1 用户模块（注册/登录）

路由：
- `POST /api/register`
- `POST /api/login`

关键文件：
- `controller/user_controller.go`
- `service/user_service.go`
- `model/user.go`

实现原理：
- 注册：先查用户名是否存在 → bcrypt 加密 → 写入 users
- 登录：按 username 查 users → bcrypt.Compare 校验密码

为什么要用 bcrypt：
- 密码不可逆保存，防止数据库泄漏导致明文暴露

错误处理特点：
- 重复用户名：返回 `code=409`
- 用户不存在/密码错误：返回 `code=401`

### 6.2 素材模块（上传/草稿/作品列表/草稿转作品）

路由：
- `POST /api/materials/upload`（multipart）
- `GET /api/materials/list?user_id=...&status=0|1`
- `POST /api/materials/work/:id`
- `POST /api/drafts/upload`（JSON，同步 URL 的草稿记录）

关键文件：
- `controller/material_upload_controller.go`
- `controller/photo_controller.go`
- `controller/sync_controller.go`
- `service/material_service.go`
- `model/material.go`

实现原理（materials/upload）：
1) 校验 `user_id`、`status`、可选 `template_id`
2) 读取 multipart 图片字段（兼容 `image` 或 `file`）
3) 校验文件后缀（jpg/jpeg/png）
4) 保存到本地文件系统：`static/uploads/u{user}/YYYYMMDD/时间戳_随机串.ext`
5) 组装对外 public URL：`http(s)://{host}/static/uploads/...`
6) 落库 materials
7) 如果传了 `template_id`，则 `IncrementTemplateUsage(template_id, 1)`

实现原理（materials/list）：
- 按 user_id + status 查询 materials

实现原理（materials/work/:id）：
- 把 materials.status 从 0 更新为 1

为什么上传要生成随机文件名：
- 避免用户原始文件名包含路径、特殊字符
- 避免重名覆盖
- 避免信息泄露（如原文件名包含隐私）

### 6.3 构图分析模块（经典构图识别：ONNX）

路由：
- `POST /api/composition/analyze`（multipart：`image`）

关键文件：
- `controller/composition_controller.go`
- `service/composition_service.go`
- `composition/composition.go`
- `models/comp_model.onnx`

实现原理（端到端）：
1) Controller 读取上传图片（字段名 `image`）并校验格式
2) Service 把图片保存到临时目录（`os.TempDir()/photo_upload`）
   - 生成安全随机文件名
   - 推理结束后删除临时文件
3) 调用 `compositionModel.Predict(tempFile)` 执行 ONNX 推理
4) 输出：构图类别列表 + 置信度

模型输入输出约定：
- 输入 Tensor shape：`1x3x224x224`
- 输出 Tensor shape：`1x9`（9 类构图）

预处理关键点（composition/preprocessImage）：
- `letterbox` 等比缩放 + 补黑边：避免强行拉伸导致构图比例失真
- 转 RGB 并归一化：使用 ImageNet mean/std

后处理关键点：
- 模型输出为 logits，代码对每个类别做 sigmoid：
  - $p = 1/(1+e^{-x})$
- 阈值策略：
  - 取 $p \ge 0.35$ 的类别
  - 如果都不达标，则返回最高概率的 1 类（避免空结果）

常见坑与已做的兜底：
- ONNX runtime 动态库版本与 onnxruntime_go 绑定不匹配 → 初始化会报 “requested API version”
  - `composition.InitCompositionModel` 对错误信息做了更友好的提示

### 6.4 模板模块（列表/热门/详情/搜索/推荐）

路由：
- `GET /api/templates/hot`
- `GET /api/templates/list`
- `GET /api/templates/detail`
- `GET /api/templates/search`
- `GET /api/templates/recommend`

关键文件：
- `controller/template_controller.go`
- `service/template_service.go`
- `service/template_recommend_service.go`
- `service/template_usage_service.go`
- `static/templates/templates.json`

#### 6.4.1 模板数据从哪里来（为什么不建 templates 表）

你们当前把模板作为“静态资源”管理：
- 图片/overlay 直接存在 `static/templates/{template_id}/...`
- 元数据集中在 `templates.json`

好处：
- 不依赖数据库迁移
- 资源可随发布版本一起升级
- 模板数量较少时非常轻量

#### 6.4.2 ListTemplates：扫描目录 + 读取 templates.json

`service.ListTemplates(dir, limit)` 的两种模式：

- 优先模式：如果 `templates.json` 里提供了 `template_id`（推荐做法）
  - 以 meta 为准组装模板
  - 支持 cover/example/overlay 三图 + guides

- 兼容模式：如果没有 meta 或 meta 不完整
  - 扫描目录里的 jpg/png/webp 文件
  - 用文件名（去后缀）作为 template_id

路径转换规则：
- meta 中写相对路径（如 `sea_bigshot_01/cover.jpg`）
- 服务端转成 `/static/templates/sea_bigshot_01/cover.jpg?v=mtime`
  - `v=` 取文件修改时间戳，简单解决缓存更新

#### 6.4.3 Hot：hot + used_count

热门接口：`ListHotTemplates`

排序分数：
- 基础：`hot`
- 叠加：`usageBoost(used_count)`
  - `usageBoost` 用对数缩放（log10），避免 used_count 过大碾压一切

#### 6.4.4 Detail：返回三图 + guides + used_count + favored

详情接口：`GetTemplateDetail`

- 找到 template_id 对应模板
- 调 `DecorateTemplates` 补充：
  - `used_count`（来自 template_usages）
  - `favored`（如果传 user_id，会查 template_favorites）

#### 6.4.5 Recommend：偏好 + 收藏 + 作品信号

推荐接口：`RecommendTemplates(userID, ...)`

信号来源：
- `preferences.tags`：用户手动选择的偏好 tags（每个 tag 权重 +1）
- `template_favorites`：收藏的模板所包含的 tags（每个 tag 权重 +3）
- `materials`：用户作品里使用过的模板（每个 tag 权重 +2）
- `template_usages.used_count`：全局使用量（对数缩放加分）
- `template.hot`：运营热度基础分

最终对每个模板计算 score 并排序。

为什么需要 include_favored：
- 有些页面希望“推荐里排除已收藏”
- 有些页面希望“推荐里包含收藏但标记 favored=true”

#### 6.4.6 Search：命中列表也按推荐排序

搜索接口：`SearchAndRecommendTemplates`

- 先按关键词 tokens 过滤出 matched
- 如果有 user_id：
  - 先生成一个大范围推荐排序（5000）
  - 把 matched 按推荐 rank 排序（更符合个性化）
- 如果无 user_id：
  - 退化为热门排序

### 6.5 收藏模块（Template Favorites）

路由：
- `POST /api/templates/favorites`
- `DELETE /api/templates/favorites`
- `GET /api/templates/favorites`

关键文件：
- `controller/template_favorite_controller.go`
- `service/template_favorite_service.go`
- `model/template_favorite.go`

实现原理：
- 收藏：插入一条 (user_id, template_id)
  - 依赖唯一键，重复收藏返回 `code=409`
- 取消收藏：按 user_id+template_id delete
  - 若 rowsAffected=0 返回 `code=404`

### 6.6 偏好模块（Preference Tags）

路由：
- `GET /api/preferences/tags?user_id=...`
- `POST /api/preferences/tags`

关键文件：
- `controller/preference_controller.go`
- `service/preference_service.go`
- `model/preference.go`

实现原理：
- tags 在 DB 中按逗号分隔字符串保存
- Get：读出后按分隔符解析为数组
- Upsert：不存在就 Create，存在就 Save

为什么用逗号分隔存储：
- 简单轻量
- tags 数量不大时够用

> 如果未来 tags 很多且需要复杂查询（包含、交集等），再考虑独立 tag 表或 JSON 字段。

## 7. VLM 模块（拍摄建议 advice / 人像姿势引导 pose）

这是你们最近新增并打通的能力：

- Go 后端对外：`POST /api/vlm/infer`
- Go 后端内部：把 multipart 转发到 Python：`POST /vlm/infer`
- Python VLM：加载 Qwen2-VL，严格输出 JSON

### 7.1 对外契约（Go：/api/vlm/infer）

路由：`POST /api/vlm/infer`（multipart/form-data）

字段：
- `task`：`advice` 或 `pose`
- 图片字段名：`image` 或 `file`（两者兼容）

返回：

```json
{"code":200,"msg":"ok","data":{...}}
```

data 的结构：

- advice：只能且必须包含 3 个 key

```json
{
  "composition": "构图优化：...",
  "focus": "焦点调整：...",
  "atmosphere": "氛围强化：..."
}
```

- pose：只能且必须包含 2 个 key

```json
{
  "pose_title": "✨ 推荐姿势：...",
  "instructions": ["...", "...", "..."]
}
```

### 7.2 Go 侧实现原理（转发 + 校验）

关键文件：
- `controller/vlm_controller.go`
- `service/vlm_service.go`

处理流程：
1) Controller 校验 `task` 是否为空
2) 读取上传图片（兼容 `image/file`）并校验后缀
3) Service 根据 `task` 调 `InferVLM`
4) `InferVLM` 根据 task 分流：
   - `InferShootingAdvice` / `InferPoseGuide`
5) `forwardToVLM`：构建 multipart 并转发到 Python
6) Go 对 Python 返回做 `json.Unmarshal`
   - advice 校验三个字段非空
   - pose 校验 title 非空 + instructions 非空

关键工程点（你们踩过的坑）：
- multipart 转发必须同步构建 body，不能用 pipe+goroutine 静默失败
- 对 loopback（127.0.0.1/localhost）禁用系统代理（否则某些环境会走代理导致超时）

### 7.3 Python VLM 实现原理（严格 JSON 输出闭环）

目录：`vlm_api/`

关键文件：
- `vlm_api/app.py`

接口：
- `GET /health`
- `GET /model/status`
- `POST /vlm/infer`

核心目标：**让输出能被 json.loads 直接解析，并且 key 完全固定**。

你们采用了“强约束 + 校验 + 修复重试”的闭环：

1) 强 System Prompt
- 明确禁止 Markdown/代码块/解释文字
- 只允许输出 1 个 JSON 对象
- key 固定（advice 3 个，pose 2 个）
- value 有固定前缀（构图优化/焦点调整/氛围强化；pose_title 必须以 ✨ 推荐姿势： 开头）
- 禁止省略号和占位符

2) JSON 抽取
- 模型偶尔会输出多余文本
- 用“括号深度扫描”抽出第一个完整 `{...}` 作为候选 JSON

3) Schema 校验（Pydantic）
- `extra=forbid`：禁止输出多余字段
- 字符串长度、数组长度等约束
- 前缀校验

4) Repair 重试
- 如果校验失败，把失败原因和示例 schema 反馈给模型，让它“只输出 JSON”重写
- 至少两轮修复策略（逐渐更强硬）

5) 并发控制
- 用 `asyncio.Semaphore(MAX_CONCURRENT)` 控制并发，避免 GPU OOM

6) 懒加载模型
- 首次请求或后台加载时才初始化 transformers 模型

Python 侧常用环境变量：
- `VLM_MODEL_ID`：默认 `Qwen/Qwen2-VL-2B-Instruct`
- `VLM_MAX_CONCURRENT`：默认 1
- `VLM_DRY_RUN=1`：不跑模型直接返回固定 JSON（快速联调）
- `VLM_TORCH_DTYPE`、`VLM_DEVICE_MAP`：控制 dtype/device

## 8. 一些“设计选择”的解释（帮助写论文/答辩）

### 8.1 为什么把“模板”做成静态资源而不是数据库表

- 模板本质是产品资源（图片+overlay+引导），变更频率低
- 静态资源更适合版本管理与 CDN/缓存
- 数据库只存“收藏/使用量/偏好”等用户行为信号

### 8.2 为什么 VLM 不直接嵌到 Go 里

- VLM 推理依赖 CUDA/torch/transformers，部署复杂
- Go 后端更适合做业务 API 与 I/O
- 通过 HTTP 抽象把推理服务解耦，未来可替换模型/机器

### 8.3 为什么严格 JSON 不能只靠 Prompt

- 大模型天然会输出解释、换行、Markdown
- UI 渲染需要强契约
- 因此必须做：抽取 JSON + schema 校验 + 失败重试

## 9. 当前项目的已知限制与注意事项

- 目前没有做登录态 token / 鉴权：客户端需要显式传 `user_id`
- CORS 允许 `*`：适合开发联调，生产应收敛域名
- 上传大小限制：Gin `MaxMultipartMemory = 10MB`
- Windows 本地编译 Go（含 onnxruntime_go）可能需要 gcc/CGO；服务器 Linux 编译通常更稳定
- 当前联调形态依赖开发机在线与 SSH 隧道稳定

## 10. 快速定位问题（排查顺序建议）

1) 服务器 Go 是否在跑：`curl http://127.0.0.1:8080/...` 或 `systemctl status`
2) 构图模型是否初始化成功：看 Go 启动日志里是否有“构图模型加载成功”
3) VLM 隧道是否通：服务器 `curl http://127.0.0.1:8000/health` 是否 200
4) Python VLM 是否可用：本地 `http://127.0.0.1:8000/model/status`
