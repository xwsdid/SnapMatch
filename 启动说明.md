# 启动说明（photo_backend：Go 后端 + Python VLM + SSH 反向隧道）

更新时间：2026-03-19

当前联调形态：
- 服务器（Linux）：运行 Go/Gin 后端（systemd 常驻，默认 `:8080`）
- 本地（Windows，有 GPU）：运行 Python VLM 推理服务（Uvicorn，`127.0.0.1:8000`）
- 连接方式：本地通过 SSH **反向端口转发**把本地 `127.0.0.1:8000` 映射到服务器 `127.0.0.1:8000`，从而让服务器 Go 能访问到本地 GPU 推理。

---

## 0. 每天必须保证的 3 个“常驻进程”

1) 本地 Windows：Python VLM（Uvicorn）常驻
2) 本地 Windows：SSH 反向隧道常驻（`ssh -N -R ...`）
3) 服务器 Linux：Go 后端常驻（systemd 服务 `photo-backend`）

三者都在，`POST http://<服务器>:8080/api/vlm/infer` 就能稳定返回 advice/pose 的严格 JSON。

---

## 1. 只需要做一次的初始化（服务器）

> 目标：把 Go 二进制、systemd、环境变量一次性配好。后续每天只需要“启动/重启”。

### 1.1 确认系统类型（可选）

`cat /etc/os-release`

### 1.2 安装编译依赖（只需一次）

构图（ONNX Runtime）功能要求 **Linux + cgo** 编译产物，因此服务器需要 gcc 工具链。

- Ubuntu/Debian：

`sudo apt-get update && sudo apt-get install -y build-essential`

- RHEL/CentOS/Alibaba Cloud Linux/OpenAnolis：

`sudo yum -y install gcc gcc-c++ make`

### 1.3 在服务器编译 Go 二进制（必须在服务器上编）

注意：不要用 Windows 编译出来的 `photo_backend_server` 覆盖服务器版本，否则很容易出现 `ONNXRUNTIME_DISABLED`（构图模块被编译期禁用）。

`cd /root/photo_backend`

建议显式开启 cgo：

`CGO_ENABLED=1 go build -o photo_backend_server .`

### 1.4 写 systemd 环境变量文件

环境文件：`/etc/photo_backend.env`

常用检查：

`sudo cat /etc/photo_backend.env`

如果从 Windows 编辑/拷贝导致 CRLF，修正一次即可：

`sudo sed -i 's/\r$//' /etc/photo_backend.env`

关键变量（当前联调形态）：
- `VLM_API_BASE_URL=http://127.0.0.1:8000`
- `PHOTO_DB_DSN=...`
- `ONNXRUNTIME_SHARED_LIBRARY_PATH=/root/onnxruntime-.../lib/libonnxruntime.so`
- `COMPOSITION_MODEL_PATH=/root/photo_backend/models/comp_model.onnx`

### 1.5 安装并启用 systemd 服务

service 文件：`/etc/systemd/system/photo-backend.service`

`sudo systemctl daemon-reload`

`sudo systemctl enable --now photo-backend`

## 2. 每次开工启动清单（推荐顺序）

### 2.1 本地 Windows 启动 Python VLM（必须）

在 VS Code 里开一个终端（PowerShell/CMD/Git Bash 都可以），启动 VLM：

1) 进入目录

`cd C:\4C_feng\photo_backend\vlm_api`

2) 直接用 venv 里的 python 启动（不需要激活 venv）

推荐启动（联调稳定版，避免 JSON 不合规 / CUDA OOM 导致前端无显示）：

PowerShell：

`$env:VLM_FAIL_OPEN="1"; $env:VLM_DEBUG="0"; $env:VLM_RETURN_META="0"; .\.venv\Scripts\python.exe -m uvicorn app:app --host 127.0.0.1 --port 8000 --log-level info`

说明：
- `VLM_FAIL_OPEN=1`：模型输出不合规或推理异常时，返回兜底 JSON（前端不黑屏）
- `VLM_DEBUG=1`：需要排查链路/耗时/是否走兜底时再打开
- `VLM_RETURN_META=1`：返回附带 `__source/__sha16`（只用于排查；给前端用建议设回 0）
- 大图下采样：`VLM_MAX_IMAGE_SIDE` 默认已更安全（避免 4K 图触发 CUDA OOM）；如需手动指定可加：`$env:VLM_MAX_IMAGE_SIDE="1280"`

3) 本地自检（保持该终端不关闭）

PowerShell（推荐，`curl` 在 PS 里通常是 `Invoke-WebRequest` 的别名，不支持 `-sS`）：

`irm http://127.0.0.1:8000/health`

如果你确实想用 curl：

`curl.exe -sS http://127.0.0.1:8000/health`

4) 重启 VLM（推荐用“先停端口再启动”，避免 WinError 10048）

先停止占用 8000 的进程：

`$conns = Get-NetTCPConnection -LocalPort 8000 -State Listen -ErrorAction SilentlyContinue; if ($conns) { $pids = $conns | Select-Object -ExpandProperty OwningProcess -Unique; foreach ($id in $pids) { Stop-Process -Id $id -Force -ErrorAction SilentlyContinue } }`

再按上面的“推荐启动”命令重新启动即可。

### 2.2 本地 Windows 建立 SSH 反向隧道（必须）

另开一个 VS Code 终端 Tab（保持该终端不关闭）：

`ssh -N -R 8000:127.0.0.1:8000 root@8.130.157.142 -p 22 -o ExitOnForwardFailure=yes -o ServerAliveInterval=30 -o ServerAliveCountMax=3`

### 2.3 服务器重启 Go 后端（systemd，必须）

`sudo systemctl restart photo-backend`

查看状态：

`sudo systemctl status photo-backend --no-pager`

查看日志：

`sudo journalctl -u photo-backend -n 200 --no-pager`

## 3. 每次都建议做的验收（服务器）

1) 验证隧道是否通（服务器访问 `127.0.0.1:8000` 应返回 uvicorn JSON）

`curl -sS http://127.0.0.1:8000/health`

2) 验证 Go → VLM advice

`curl -s -F "task=advice" -F "file=@/root/photo_backend/static/templates/sea_bigshot_01/example.jpg" http://127.0.0.1:8080/api/vlm/infer`

3) 验证 Go → VLM pose

`curl -s -F "task=pose" -F "file=@/root/photo_backend/static/templates/sea_bigshot_01/example.jpg" http://127.0.0.1:8080/api/vlm/infer`

期望：两条都返回 `{"code":200,...}`。

## 4. 停止项目（下班/不联调时）

### 4.1 本地 Windows

- 停止 Python VLM：在运行 uvicorn 的窗口按 `Ctrl + C`
- 停止 SSH 隧道：在运行 ssh 的窗口按 `Ctrl + C`

### 4.2 服务器 Linux

- 停止 Go 服务：`sudo systemctl stop photo-backend`

## 5. 常见问题速查

### 5.1 服务器 `curl http://127.0.0.1:8000/health` 不通

说明：SSH 反向隧道没建好或断了。

按顺序检查：
- 本地 VLM 是否在跑（本地 `http://127.0.0.1:8000/health` 是否 200）
- 本地 ssh 窗口是否还在（是否退出/报错）
- 服务器是否已有进程占用 8000（排查：`ss -ltnp | grep :8000`）

### 5.2 `/api/vlm/infer` 500 或超时

常见原因：
- `VLM_API_BASE_URL` 写错（联调形态应是 `http://127.0.0.1:8000`）
- SSH 隧道断开
- VLM 首次加载模型耗时较长（可先请求一次 `/health` 或等待首次加载完成）
- 本地 VLM CUDA OOM（常见于 3K/4K 大图 + 连续点击触发多次请求；VLM 会直接 500，前端不显示）

排查/处理建议：
- 先确认本地 VLM 直接接口是否 200：`curl.exe -sS -F "task=advice" -F "file=@C:\\4C_feng\\photo_backend\\static\\templates\\sea_bigshot_01\\example.jpg" http://127.0.0.1:8000/vlm/infer`
- pose 同理：`curl.exe -sS -F "task=pose" -F "file=@C:\\4C_feng\\photo_backend\\static\\templates\\sea_bigshot_01\\example.jpg" http://127.0.0.1:8000/vlm/infer`
- 如出现 OOM：确保按 2.1 的“推荐启动”运行（`VLM_FAIL_OPEN=1` + 下采样），并避免前端连续多次点击同一按钮

查看 Go 服务日志：

`sudo journalctl -u photo-backend -n 200 --no-pager`

### 5.3 本地启动 uvicorn 报 WinError 10048（8000 被占用）

1) 查占用：

`Get-NetTCPConnection -LocalAddress 127.0.0.1 -LocalPort 8000 -State Listen | Format-Table -AutoSize`

2) 查命令行：

`Get-CimInstance Win32_Process -Filter "ProcessId=xxxx" | Select-Object ProcessId,Name,CommandLine | Format-List`

3) 结束进程：

`Stop-Process -Id xxxx -Force`

### 5.4 服务器日志出现 `ONNXRUNTIME_DISABLED`（构图功能不可用）

现象：
- `journalctl -u photo-backend` 里出现：`构图模型初始化失败：ONNXRUNTIME_DISABLED`。

原因：
- 当前服务器运行的 `photo_backend_server` 是“编译期未启用 ONNX Runtime”的构建产物（常见：用 Windows/关闭 cgo 编译后上传到 Linux）。

处理：
1) 确保服务器安装了 gcc 工具链（见 1.2）
2) 在服务器 `/root/photo_backend` 重新编译：`CGO_ENABLED=1 go build -o photo_backend_server .`
3) 重启服务：`sudo systemctl restart photo-backend`

## 6. 生产形态提示（可选）

当前“服务器无 GPU + 反向隧道”适合开发联调。

若要长期稳定运行，建议把 Python VLM 部署到一台常驻 GPU 机器上，并把服务器 `/etc/photo_backend.env` 的 `VLM_API_BASE_URL` 指向那台 GPU 机器内网地址（例如 `http://10.0.0.12:8000`），不再依赖开发机与 SSH 隧道。
